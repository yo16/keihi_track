# ケイトラ（keihi_track）要件定義書

## 1. プロジェクト概要

### 1.1 目的

小規模組織向けの経費管理システム。経費の申請・承認ワークフローをWebアプリケーションとして提供する。

### 1.2 システム名

- 正式名称: 経費トラック
- 略称: ケイトラ（keihi_track）

### 1.3 想定規模

- 同時利用ユーザー: 数人〜数十人程度
- 組織単位での利用

## 2. 技術スタック

| 項目 | 技術 |
|------|------|
| フロントエンド | Next.js + TypeScript |
| データベース | Supabase (PostgreSQL) |
| 認証 | Supabase Auth |
| ファイルストレージ | Supabase Storage |
| メール送信 | Resend（Supabase AuthのカスタムSMTP + メール通知用API） |
| デプロイ | Vercel |
| UI言語 | 日本語のみ |

## 3. ユーザーとロール

### 3.1 ロール定義

| ロール | 説明 | 権限 |
|--------|------|------|
| 管理者 | 組織を作成した人。使用者・承認者の全機能を兼ねる | ユーザー管理、ロール変更、経費申請、承認 |
| 承認者 | 経費を承認する人。使用者の全機能を兼ねる | 経費申請、承認、一覧確認、CSV出力 |
| 使用者 | 経費を申請する人 | 経費申請、自分の申請一覧確認 |

ロールの包含関係: 管理者 ⊃ 承認者 ⊃ 使用者

### 3.2 ロール変更ルール

- 管理者が他ユーザーのロールを変更できる
- 変更可能なパターン: 使用者 ⇔ 承認者、任意のロール → 管理者、管理者 → 任意のロール
- **バリデーション**:
  - 組織に最低1人の管理者が必要（管理者が0人になる変更は不可）
  - いかなるロールでも、自分自身のロールを自分で変更することはできない

### 3.3 ユーザーの識別

- ユーザーはメールアドレスで一意に識別される
- **1つのメールアドレスは1つの組織にのみ所属可能**（DBレベルで `organization_members.user_id` にユニーク制約を設ける）

### 3.4 ユーザーの削除

- 論理削除とする（過去の申請データとの整合性を維持するため）

## 4. 認証

### 4.1 認証方式

- Supabase Authによるメールアドレス＋パスワード認証

### 4.2 ログイン

- トップページ（`/`）にログインフォーム（メールアドレス＋パスワード）を配置
- ログイン成功後、ユーザーの所属組織をサーバーで自動特定し、ダッシュボードへ遷移
- 組織IDの入力は不要（1メール=1組織のため、システムが自動判定）

### 4.3 アカウント作成とパスワード設定

- 組織作成時: ユーザーが自分でメールアドレスとパスワードを入力してサインアップする（サーバーサイドでAdmin APIを使用し、メール確認をスキップ）
- ユーザー招待時: 管理者がメールアドレスとロールを指定し、Supabase Authの招待メール機能（`inviteUserByEmail`）で招待メールを送信する。招待されたユーザーは、メール内のリンクからパスワードを自分で設定する

## 5. 組織管理

### 5.1 組織作成フロー

1. ユーザーがトップページ（`/`）から**作成パスワード**・メールアドレス・パスワード・組織名・表示名を入力して組織を新規作成
2. サーバーサイドで作成パスワードを環境変数 `ORG_CREATION_PASSWORD` と照合し、一致しなければ拒否する
3. サーバーサイドでアカウント作成（Admin API、メール確認スキップ）と組織作成を一括実行
4. 組織を作成したユーザーが自動的にその組織の管理者ロールとなる
5. クライアントでログインし、`/dashboard` へ遷移

#### 作成パスワードによるゲート（暫定仕様）

- 組織の無制限な作成を防ぐため、組織作成時に「作成パスワード」の入力を必須とする
- 作成パスワードはサーバーサイドの環境変数 `ORG_CREATION_PASSWORD` に設定された文字列と単純一致で検証する
- 環境変数が未設定の場合、組織作成は不可とする（安全側に倒す）
- この仕組みは簡易的なゲートであり、将来的に廃止または別の認可方式に置き換える可能性がある

### 5.2 ユーザー招待フロー

1. 管理者が `/admin/members` でメールアドレス、表示名、ロールを指定してユーザーを招待する
2. Supabase Authの `inviteUserByEmail` により、招待メールが自動送信される
3. 管理者には「招待メールを送信しました。{メールアドレス}にメールが届きます。」と表示される
4. 招待されたユーザーがメール内のリンクをクリックすると、`/auth/callback` 経由で `/set-password` に遷移する
5. パスワードを設定すると、トップページ（`/`）に「パスワードが設定されました。ログインしてください。」と表示される
6. メールアドレスとパスワードでログインし、`/dashboard` へ遷移する

### 5.3 メール送信基盤

- メール送信はSupabase Authの組み込み機能を利用する
- カスタムSMTPプロバイダとしてResendを使用する（ダッシュボードで設定）
- メールテンプレートはSupabaseダッシュボードでカスタマイズ可能

## 6. 経費申請

### 6.1 入力項目

| 項目 | 必須 | 説明 |
|------|------|------|
| 金額 | 必須 | 日本円（整数） |
| 用途 | 必須 | テキスト入力 |
| 使用日 | 必須 | 経費を使用した日付 |
| レシート写真 | 必須 | 1申請につき1枚 |
| コメント | 任意 | 補足情報のテキスト入力 |

### 6.2 レシート写真の処理

- **サムネイル画像**: 長辺300px程度に縮小して保存。システム上の表示にはこちらを使用
- **オリジナル画像**: 元のサイズのまま別途保存。保存まではシステムスコープ内、その後の利用はスコープ外
- **ストレージ**: Supabase Storageに格納
- **共有範囲**: URLを知っている人がアクセス可能な状態（公開URL）

### 6.3 経費ステータス

| ステータス | 説明 |
|-----------|------|
| 申請中 | 使用者が申請を提出した状態 |
| 承認済み | 承認者が承認した状態 |
| 却下（修正中） | 承認者が却下した状態。使用者が修正して再申請可能 |
| 削除 | 使用者が取り下げた状態（論理削除） |

### 6.4 ステータス遷移

```
[申請中] → [承認済み]       （承認者が承認）
[申請中] → [却下（修正中）]  （承認者が却下 ※コメント必須）
[申請中] → [削除]           （使用者が取り下げ）
[却下（修正中）] → [申請中]  （使用者が修正して再申請）
```

- 却下時、承認者はコメント（却下理由）を必ず付ける
- 却下された申請は、使用者が内容を編集して再申請できる（経費IDは変わらない）
- レシート写真も再申請時に差し替え可能
- 「申請中」ステータスの申請は、使用者が取り下げ（論理削除）できる

### 6.5 承認ルール

- 承認者ロールまたは管理者ロールを持つユーザーが承認可能
- **自分自身の申請は原則として承認できない**
  - ただし、組織内のアクティブな承認権限者（管理者+承認者）が1人のみの場合は、自己承認を許可する
- 同じ組織内の承認者であれば誰でも承認可能

## 8. 通知

### 8.1 アプリ内通知（必須）

- ヘッダーにベルアイコン＋未読バッジを表示
- 通知一覧画面で詳細を確認可能

| 通知タイミング | 通知対象 | 内容 |
|---------------|---------|------|
| 新規申請時 | 承認者 | 「{使用者名}が経費申請を提出しました」 |
| 承認時 | 使用者 | 「経費申請が承認されました」 |
| 却下時 | 使用者 | 「経費申請が却下されました」 |
| 再申請時 | 承認者 | 「{使用者名}が経費申請を再提出しました」 |

### 8.2 メール通知

- 経費ステータス変更時に、対象ユーザーのメールアドレスへ通知メールを送信する
- メール送信はAPI Routeから、Resend APIを使用して送信する
- 通知タイミング・対象はアプリ内通知（8.1）と同一

## 9. 経費一覧・CSV出力（承認者向け）

### 9.1 一覧表示

- 全ステータスの経費を表示
- フィルター機能:
  - 日付（from, to）
  - ステータス

### 9.2 CSV出力

- 一覧の左端にチェックボックスを配置
- チェックされたレコードのみCSV出力対象
- フィルターにより非表示のレコードはチェックオフとして扱う（出力対象外）
- 出力項目（固定）:

| CSV項目 |
|---------|
| 金額 |
| 用途 |
| 使用日 |
| レシート写真URL（Supabase StorageのURL） |
| コメント |
| 申請者 |
| 申請日時 |
| 承認者 |
| 承認日時 |
| 却下者 |
| 却下日時 |

## 10. スコープ外

- 紙のレシートの受け渡し管理
- オリジナルレシート写真の保存後の利用・管理
- 多言語対応
- ネイティブアプリ
- 1メールアドレスでの複数組織所属

## 11. 非機能要件

### 11.1 セキュリティ

- Supabase Authによる認証
- Row Level Security (RLS) による組織単位のデータ分離
- パスワードは Supabase Auth が管理（平文保存しない）

### 11.2 パフォーマンス

- レシート写真はサムネイル（長辺300px）を一覧表示に使用し、表示速度を確保

### 11.3 データ保全

- ユーザー・経費ともに論理削除を採用し、データを物理的に削除しない
