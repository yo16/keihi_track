# タスク記述フォーマット

仕様書からBeadsタスクに分解する際の、descriptionの記述フォーマット。
タスクを作成するAIと、タスクを実装するcoderエージェントの両方がこのフォーマットに従う。

## フォーマット

```markdown
## 実装内容
- このタスクで実装する機能の具体的な振る舞いを記述する
- ユーザーから見た動作、またはシステムの内部処理を明確にする
- 曖昧な表現（「適切に処理する」等）を避け、具体的に書く

## 対象箇所
- 作成・変更するファイルやディレクトリのヒント
- 既存コードとの関連（どのモジュールに属するか等）
- 新規ファイル作成が必要な場合はその旨を明記する

## 受け入れ条件
- このタスクが完了したと判断できる条件を列挙する
- 各条件は検証可能な形で書く（「〜できること」「〜が返ること」等）

## テスト観点
- 正常系: <テストすべき正常動作>
- 異常系: <テストすべきエラーケース>
- 境界値: <テストすべき境界条件>（該当する場合）
```

## 記述のガイドライン

- **1タスク = 1機能単位**: タスクが大きすぎると実装・テストが複雑になる。迷ったら分割する
- **仕様書への参照**: `docs/specification.md` の該当セクションがあれば「仕様書 §X を参照」と記載する
- **依存関係の明示**: 他のタスクの成果物（関数、型定義等）を前提とする場合は、Beadsの依存関係（`bd dep add`）で設定する
- **coderが判断に迷わないこと**: descriptionを読んだだけで、何をどこに作ればよいか分かる粒度で書く
