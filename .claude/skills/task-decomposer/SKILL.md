# 仕様書 → Beadsタスク分解

仕様書（`docs/specification.md`）を読み、実装可能な粒度のBeadsタスクに分解・登録する。

## 処理フロー

### 1. 仕様書の読み込み
- `docs/specification.md` を読み、全体像を把握する
- `CLAUDE.md` を読み、プロジェクトの技術スタック・ディレクトリ構成を把握する
- `knowledge/` ディレクトリが存在する場合、該当する技術制約を確認する

### 2. タスクへの分解
- 仕様書の機能単位でタスクに分解する
- 各タスクは `references/task-template.md` のフォーマットに従って記述する
- 以下の観点で分解の粒度を判断する:
  - **1タスク = 1機能単位**: coderエージェントが1回で実装・テストできる大きさ
  - **依存関係が明確**: タスク間の前後関係が分かる
  - **テスト可能**: 単体でテストが書ける

### 3. 依存関係の設定
- タスク間の依存関係を分析する
- 型定義・ユーティリティなど、他タスクの前提となるものは先に実行されるよう依存を設定する
- 循環依存が生じないよう注意する

### 4. Beadsへの登録
- `bd create` でタスクを登録する
- descriptionは複数行になるため、`tmp/` に一時ファイルを書き出し `--body-file` で渡す
- 依存関係は `bd dep add` で設定する
- 登録後、一時ファイルを削除する

### 5. 登録結果の報告
- 作成したタスクの一覧を表示する（ID、タイトル、依存関係）
- 全体の実行順序（トポロジカルソート結果）を表示する
- ユーザーに確認を求める

## 分解の原則

### やるべきこと
- 仕様書に書かれている機能を漏れなくタスク化する
- 各タスクの受け入れ条件を具体的に書く
- テスト観点（正常系・異常系・境界値）を必ず含める
- 対象箇所（ファイル・ディレクトリ）のヒントを書く

### やるべきでないこと
- 仕様書にない機能を勝手に追加しない
- 1タスクに複数の機能を詰め込まない
- 曖昧な受け入れ条件（「適切に動作すること」等）を書かない

## 参照ファイル
- `references/task-template.md` : タスクのdescription記述フォーマット
