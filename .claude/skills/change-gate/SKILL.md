# 変更管理ゲート

ユーザーの問題報告・改善要望・変更依頼を検知し、正式な変更管理プロセスに乗せるためのゲートスキル。

## 目的

開発中に発生した問題や改善要望を、即座にコード修正するのではなく、分析→承認→Beads登録→実装という規律あるプロセスで処理する。
Beadsタスクとして記録することで、「何が問題だったのか」「何がきっかけで発覚したのか」という背景情報を残す。

## トリガー条件

以下のようなユーザー発言を検知した場合に起動する:

- **問題報告**: 「動かない」「おかしい」「エラーが出る」「バグ」「失敗する」「期待通りでない」「壊れた」
- **改善要望**: 「改善したい」「もっと〇〇にしたい」「使いにくい」「見づらい」「遅い」
- **変更依頼**: 「変えてほしい」「修正して」「直して」「〇〇を変更して」「〇〇を追加して」「〇〇を削除して」
- **リファクタリング**: 「リファクタ」「整理して」「構造を変えたい」「きれいにして」

## 起動しない条件

以下の場合はこのスキルを起動しない:

- 質問・確認: 「この関数は何をしている？」「設計の意図は？」「なぜこうなっている？」
- 初期構築中のタスク実行: `/batch-start` や `/batch-task-execute` の実行中
- 設計フェーズ: `/batch-design` の実行中
- 既に `/batch-change` プロセスが進行中

## 処理フロー

### 1. ユーザーの発言内容を要約する
- 何について言っているのか（対象の機能・画面・コンポーネント）
- どういう問題・要望なのか（現象・期待との差分）

### 2. ユーザーに確認する
以下のように確認する:

```
これは問題報告（または改善要望）として、変更管理プロセスに乗せますか？

【検知した内容】
- 対象: 〇〇
- 内容: 〇〇

変更管理プロセスでは、原因分析 → 対応方針の提示 → Beads登録 → 実装 の順で進めます。

→ /batch-change を実行しますか？ (yes/no)
```

### 3. ユーザーの回答に応じて分岐
- **yes**: `/batch-change` コマンドの処理フローに移行する
- **no**: 通常の対話を継続する（ただし、直接コード修正は行わず、情報提供・分析にとどめる）

## 重要な制約
- **このスキルが起動された時点で、コードの直接修正は行わない**
- ユーザーの承認なしに `/batch-change` を自動実行しない
- 「軽微だから」という理由でこのスキルをスキップしない
