# /batch-design

DB設計とDB操作API設計を行うコマンド。
仕様書に基づき、db-designerが設計を行い、必要に応じてsupabase-specialistと連携する。
設計結果は `docs/db-design.md` に出力する。

## 引数
- `$ARGUMENTS` : 設計対象の補足説明（任意）

## 使用するサブエージェント
- `db-designer` : DB設計・API設計の主担当
- `supabase-specialist` : Supabase利用時の固有機能に関する助言・設計

## 処理フロー

### 1. 要件の理解（Analyze）
- `docs/specification.md` を読み込み、アプリケーション仕様を理解する
- `CLAUDE.md` を読み込み、プロジェクトルールを理解する
  - 特に「DBサービス」「DB操作アーキテクチャ」「API技術スタック」「テスト方針」「環境変数」セクションを確認する
  - これらが未記載の場合は、ステップ2で合わせてユーザーに確認し、設計完了後に `CLAUDE.md` への追記を案内する
- $ARGUMENTS が指定されている場合は、その内容も考慮する
- 以下の観点で仕様を分析する:
  - 管理するデータエンティティは何か
  - エンティティ間のリレーションはどうか
  - どのようなクエリパターンが想定されるか
  - データ量の増加傾向はどうか
  - セキュリティ要件は何か

### 2. 対話的な要件探索（Explore）
- 仕様書だけでは不明確な点がある場合、ユーザーに確認する
- 確認すべき観点（`CLAUDE.md` に記載済みの項目はスキップする）:
  - 使用するDBサービス（Supabase / その他）
  - API技術スタック（Next.js API Routes / Express / Hono 等）
  - DB操作関数の配置場所（デフォルト: `src/lib/db/`）
  - テスト方針（モック使用 / テスト用DB接続）
  - 認証・認可の要件
  - データの保持期間・削除ポリシー
  - 想定されるデータ量・アクセス頻度
  - 既存データの移行要否
  - バックアップ要件
- ユーザーの回答を踏まえて設計方針を決定する

### 3. DB設計（Design） → `db-designer`
- テーブル設計を行う:
  - エンティティの抽出と定義
  - カラム定義（型、制約、デフォルト値）
  - リレーション設計（外部キー、中間テーブル）
  - 正規化レベルの判断
  - インデックス設計
- DB操作API設計を行う:
  - エンドポイント一覧
  - 各エンドポイントのリクエスト/レスポンス仕様
  - エラーハンドリング方針
  - ページネーション・フィルタリング仕様

### 4. Supabase固有の設計（Supabase利用時） → `supabase-specialist`
- 使用するDBサービスがSupabaseの場合にのみ実行する
- `db-designer` の設計をベースに、以下を検討:
  - RLS（Row Level Security）ポリシーの設計
  - Supabase Auth連携の設計
  - Storage利用の設計（必要な場合）
  - Realtime機能の適用判断（必要な場合）
  - Edge Functions活用の検討（必要な場合）
  - Supabase固有の機能による設計の最適化提案
- `db-designer` の設計に対するフィードバックを提供し、必要に応じて設計を修正する

### 5. 設計の検証（Validate）
- 設計結果を以下の観点で検証する:
  - データ整合性: 制約とリレーションは十分か
  - セキュリティ: アクセス制御と保護は適切か
  - パフォーマンス: インデックスとクエリパターンは最適か
  - メンテナンス性: 命名規約とマイグレーション計画は明確か
  - API設計: RESTful原則に準拠しているか、エラーケースは網羅されているか
- 問題があれば設計を修正する

### 6. 設計書の出力（Document）
- `docs/db-design.md` に設計結果を出力する
- 設計書のフォーマットは `db-designer` エージェントの「成果物」セクションに従う

### 7. 環境変数の定義
- `.env.sample` に必要な環境変数を記載する
- DB接続情報、APIキー等の環境変数名を定義する（値は空にする）
- Supabase利用時は、Supabase固有の環境変数も含める

### 8. ユーザーへの報告
- 設計の概要をユーザーに報告する:
  - テーブル数、主要エンティティ
  - APIエンドポイント数
  - セキュリティ方針の概要
  - 特記事項・トレードオフの説明
- `docs/db-design.md` の確認を依頼する
- `CLAUDE.md` にDB関連セクションが未記載の場合、設計で決定した内容に基づく追記案を提示する
- 設計の承認後、タスク分解（task-decomposer）に進むよう案内する
